1. Estrutura e Diagnóstico Inicial dos Dados (RAW)

O dataset original (train.csv) possuía:

  • 9.800 linhas e 18 colunas 1
  • Datas no formato DD/MM/YYYY, não padronizadas para análises
  • Ausência de métricas temporais derivadas
  • 11 valores ausentes em Postal Code 1
  • Colunas com forte dependência semântica entre si (cliente, pedido, categoria, produto)
  • Nenhuma coluna própria para chave temporal (date key)

O banco também apresentava limitações comuns em dados operacionais:

  • Falta de colunas calculadas necessárias a análises logísticas (ex.: atraso de envio)
  • Dados padronizados de forma inconsistente (datas em string, meses não identificáveis por hierarquia)
  • Ausência total de mapeamento temporal para modelos BI

Apesar disso, não foram identificadas linhas duplicadas (0 duplicatas) e todas as chaves únicas estavam preservadas 1.

2. Objetivo da Limpeza e Padronização

O processo de tratamento teve como metas:

  • Padronizar datas para ISO (YYYY-MM-DD) para garantir integridade temporal.
  • Criar colunas analíticas para uso no Power BI:
  • Ano do pedido
  • Mês do pedido (YYYY-MM)
  • Chaves de data para modelagem (Order Date Key, Ship Date Key)
  • Criar métricas operacionais:
  • Atraso de envio em dias (Ship Delay (days))
  • Garantir rastreabilidade mantendo todas as colunas originais.
  • Gerar um dataset analítico limpo (clean) para visualização.

3. Processos de Tratamento Aplicados

O dataset tratado (train_clean.csv) contém 9.800 linhas e 23 colunas, com 5 novas colunas criadas durante o processo de limpeza 2.

3.1. Conversão e padronização de datas

As colunas:

  • Order Date
  • Ship Date

Foram convertidas de string para o padrão ISO YYYY-MM-DD.

3.2. Criação de colunas analíticas

Foram geradas as seguintes colunas adicionais:

Nova Coluna Descrição
Ship Delay (days) Diferença entre envio e pedido em dias
Order Year Ano extraído da data do pedido
Order Month Mês analítico no formato YYYY-MM
Order Date Key Chave de data numérica para modelagem
Ship Date Key Chave de data numérica para modelagem

Essas cinco colunas são exatamente as encontradas no dataset limpo 2.

3.3. Correções estruturais

  • Padronização do formato de datas
  • Normalização de campos textuais
  • Validação de integridade das chaves (Row ID, Order ID, Product ID) – todas consistentes 2
  • Verificação de somatório de vendas (igual no raw e no clean)

→ 2.261.536,78, valor confirmado nas duas bases 12.

7. Análise Técnica dos Resultados

A partir do dataset já limpo:

  • Tecnologia representa o maior volume de vendas, com mais de 36% do faturamento total.
  • O atraso médio indica um processo logístico estável, variando entre 0 e 7 dias.
  • A distribuição temporal permite observar sazonalidade e picos de vendas em meses específicos.
  • A consistência dos dados permitiu cálculos confiáveis de indicadores operacionais e estratégicos.

8. Conclusão Técnica

O trabalho executado garantiu:

  • Rastreabilidade total entre dado bruto e modelo analítico
  • Padronização e qualidade do fluxo de dados
  • Construção de métricas essenciais para tomada de decisão
  • Modelo confiável para criação de dashboards analíticos
  • Dashboard interativo publicado com indicadores estratégicos validados

O resultado final é um pipeline completo, sólido e profissional, que segue princípios aplicados em times de dados reais.
